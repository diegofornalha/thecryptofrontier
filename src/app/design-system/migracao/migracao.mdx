# Plano de Migração para shadcn/UI

- Consolidamos e organizamos variáveis CSS
- Configuramos tema claro/escuro
- Definimos padrões de tipografia, cores e espaçamentos
- Instalamos componentes base do shadcn/ui
- Criamos uma página de documentação (design system)

## Fase 1 - Próximos Passos

### Componentes a serem migrados:

#### Componentes de Blocos
- [x] Bloco de imagem
- [x] Bloco de formulário
- [x] Bloco de pesquisa
- [x] Bloco de título
- [x] Bloco de vídeo


### Priorização

1. Componentes de UI básicos - são a base para todos os outros
2. Componentes de Layout - definem a estrutura do site
3. Componentes de Blocos - conteúdo específico

### Abordagem de Migração

Para cada componente:

1. Identificar o componente atual e suas variantes
2. Mapear para equivalentes do shadcn/ui
3. Criar versão com shadcn/ui mantendo a mesma API quando possível
4. Testar o componente em isolamento
5. Implementar em uma página de teste
6. Substituir gradualmente no restante do site

### Considerações

- Manter retrocompatibilidade durante a transição
- Documentar todos os componentes novos na página do design system
- Criar testes para garantir que a funcionalidade seja mantida
- Atualizar os estilos globais conforme necessário

## Métricas de Sucesso

- **Performance:** Redução de 20% no tempo de carregamento
- **Manutenibilidade:** Redução de 30% no tempo de desenvolvimento de novos componentes
- **Consistência:** 100% dos componentes seguindo os tokens do design system
- **Acessibilidade:** Conformidade WCAG AA para todos os componentes

## Estratégia de Deprecação

1. Marcar componentes antigos como deprecated com aviso no código
2. Fornecer documentação de migração para cada componente
3. Definir prazo para remoção total (3 meses após conclusão da Fase 3)
4. Remover componentes antigos após o prazo

## Gestão de Dependências

- Manter ambos os sistemas de componentes funcionando em paralelo durante a transição
- Garantir que não haja conflitos de estilos ou comportamentos
- Assegurar compatibilidade com versões existentes do site

## Estratégia para Componentes Customizados

Para componentes sem equivalente direto no shadcn/ui:
1. Avaliar se podem ser construídos com primitivos do shadcn/ui
2. Desenvolver componentes customizados seguindo os mesmos padrões
3. Documentar esses componentes específicos no design system
